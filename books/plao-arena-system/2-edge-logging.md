# この章の立ち位置

第1章では **有効受信機体数の時間積算（AUC）**
を評価指標として採用しました。\
この指標が意味を持つためには、次の条件が満たされている必要があります。

-   すべての時刻でログが欠けていない\
-   `n_used` が一貫した定義で算出されている\
-   ログが破損していない

つまり **評価の信頼性 = ログ基盤の信頼性** です。

統計モデルは後からいくらでも変えられます。\
しかし **過去のログは一度失われると二度と復元できません。**

そのため ARENA システムでは

    統計より先にログ基盤を設計する

という順序を採用しています。

言い換えると

> 正しく測る前に、正しく貯めなければならない。

本章では、エッジロガー **PLAO** の設計思想を説明します。

# Failure‑First：エッジは壊れる前提で設計する

PLAO は Raspberry Pi 上で動作します。\
クラウドのマネージドサービスではありません。

エッジ環境では次のような問題が普通に起きます。

-   突然の電源断
-   SDカードの摩耗
-   OOM Kill
-   プロセスのハング

これらは例外ではなく **日常的な故障モード** です。

そのため PLAO の設計は

    正常系をどれだけ速くするか

ではなく

    異常時に何を守るか

から始まっています。

守る対象は **過去ログの完全性** です。

新しいデータの取りこぼしはある程度許容できます。\
しかし **すでに記録したログの破損** は許容できません。

なぜなら AUC は **過去ログ全体** に依存するからです。

---

# なぜデータベースを使わないのか

PLAO は **Append‑only JSONL** を採用しています。\
SQLite すら使用していません。

理由は3つあります。

## 1. 破損範囲を限定できる

データベースはインデックスを持ちます。

そのため異常終了時には

-   WALリカバリ
-   整合性確認
-   DB修復

などの処理が必要になる可能性があります。

一方 JSONL では電源断が起きても影響は

    最後に書き込み中だった1行

にほぼ限定されます。

つまり

> 壊れたときにどこまで壊れるか

を **フォーマットの設計で制限** できます。

---

## 2. ログの透明性

JSONL は単なるテキストです。

例えば

``` bash
cat log.jsonl
jq .
```

のような標準ツールでその場で確認できます。

ログの透明性は **信頼性の重要な要素** です。

---

## 3. 解析との疎結合

PLAO の役割は

    ログを書くだけ

です。

解析や統計処理は **ARENA** が担当します。

JSONL は

    1行 = 1イベント

という構造を持つため **時系列ログと非常に相性が良い** フォーマットです。

また行単位で独立しているため

    書き込み中のファイル

を rsync でコピーしても構造が壊れません。

---

# スキーマとバージョニング

各レコードには `schema_ver` を持たせています。

例：

    {"type":"pos","schema_ver":1,"ts":1700000000.1,"hex":"abc123","lat":12.345}

JSONL では

    各行が自己記述的

という性質があります。

これにより

-   フィールド追加
-   スキーマ変更
-   バージョン混在

が起きても安全に処理できます。

---

# fsync のトレードオフ

`fsync` を呼ぶとデータ耐久性は上がります。\
しかし **SDカード寿命** は短くなります。

PLAO では

    --fsync-every

オプションで運用者が調整できます。

    耐久性
    vs
    ストレージ寿命

のバランスを **環境ごとに決められる設計** にしています。

---

# TTL によるメモリ制御

Raspberry Pi を 24/7 運用するためには\
メモリを一定範囲に保つ必要があります。

PLAO では

    一定時間更新のない機体

を削除することで **Bounded Memory** を実現しています。

これにより

-   メモリ上限を保証
-   再現可能な計算結果

を両立できます。

---

# 転送機能を PLAO に持たせない理由

ログ転送は PLAO ではなく

    rsync

に任せています。

理由は単純です。

> 転送は必ず失敗するから。

そのため PLAO の責務は

    ローカルに正しく書くこと

だけに限定します。

転送の複雑さは **枯れたツール** に任せる方が安全です。

---

# まとめ

「何を測るか」を間違えれば統計は無意味です。

しかし

**正しく貯まっていないログで行う統計は、それ以上に危険です。**

評価指標の信頼性は\
ログ基盤の信頼性によって支えられています。

壊れやすい環境では

    壊れないシステム

ではなく

    壊れても被害を最小化する構造

を選ぶべきです。

この「誠実な証人」としてのログ基盤が整って初めて\
高度な統計解析が可能になります。

**ノイズと構成変更の効果を統計的に分離する方法**
