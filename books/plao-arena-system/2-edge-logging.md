## この章の立ち位置

第1章では、評価指標として **有効受信機体数の時間積算（AUC）** を採用しました。この指標を成立させるには絶対的な前提条件があります。

- すべての時刻でログが欠けないこと
- $n_{\\text{used}}$ が一貫した定義で算出されること
- ログが破損していないこと

つまり、評価指標の信頼性は、その上流にあるログ基盤の信頼性に完全に依存します。**「正しく測る前に、正しく貯めなければならない」**のです。本章では、エッジロガー **PLAO** の設計判断を解説します。

## Failure-First：エッジは壊れる前提で設計する

PLAOが動くのは Raspberry Pi です。クラウドのマネージドサービスではありません。突然の電源断、SDカードの摩耗、メモリ不足（OOM Kill）、プロセスのハング。これらはエッジ環境では「例外」ではなく、現実に起こりうる「日常」です 。

PLAOの設計は「正常系をどう速くするか」ではなく、**「異常時に何を守るか」** から始まっています 。守るべきものは、**過去に記録したログの完全性** です。新しいデータの取りこぼしは一定範囲で許容できても、すでに書き込んだデータの破損は許容できません。AUCの計算は過去ログ全体に依存し、破損は「評価の土台」を崩すからです。

---

## なぜデータベースを使わないのか

PLAOの記録形式は **Append-only JSONL**（追記専用テキストファイル）です。SQLiteすら使わない理由は三つあります。

### 1. 破損時の影響を限定する
データベースはインデックスを含みます。異常終了時にリカバリが必要になったり、整合性確認が運用タスクとして発生しやすい性質があります。
一方、JSONLで電源断が起きた場合、影響は基本的に「最後に書き込み中だった1行」に閉じます 。最悪でも1レコードの損失で済み、それ以前のレコードは無傷で読めます。**“壊れたときにどこまで壊れるか” をファイル形式の選択で縛る。** これがエッジにおける強い利点になります。

### 2. 検証の透明性が高い
JSONLは単なるテキストです。`cat` や `jq` といった標準ツールで、ログをその場で即座に確認できます。ログの透明性は、信頼性に直結します。

### 3. 解析と疎結合にできる
PLAOは「書く」だけで、読み出しや集計は母艦（ARENA）が担います。JSONLは行単位で独立しているため、書き込み中のファイルを `rsync` で部分取得しても構造が壊れません。

## スキーマとバージョニング

各レコードに `schema_ver` を持たせています。

```json
{"type":"pos","schema_ver":1,"ts":1700000000.1,"hex":"abc123","lat":12.345, ...}
```

JSONLでは「各行が自己記述的である」性質を利用します。これにより、バージョンが混在したファイルでも、ARENA側で安全にパースできる柔軟性を確保しています。

---

## fsync のトレードオフ：寿命か、データか

物理ディスクへの同期（fsync）を呼べばデータは守られますが、SDカードの寿命を縮めます。PLAOでは、この判断をユーザーに委ねる設計にしています。

```python:src/plao/io.py
# src/plao/io.py より抜粋
def write(self, record: dict) -> None:
    self._fh.write(json.dumps(record, ensure_ascii=False) + "\n")
    self._count += 1
    if self._count % self._fsync_every == 0:
        self._fh.flush()
        os.fsync(self._fh.fileno()) # 明示的なディスク同期
```


`--fsync-every` オプションで、運用者が自分の環境に合わせてリスクをコントロールできる形にしています。

## TTL によるトラック管理：Bounded Memory の保証

ラズパイで24/7運用を続けるには、メモリ使用量を一定に保つ（Bounded Memory）ことが不可欠です。

```python:src/plao/cli_collect.py
# src/plao/cli_collect.py より抜粋
def _cleanup_tracks(self, ts: float) -> None:
    for hexid, tr in list(self.tracks.items()):
        # 一定時間（TTL）更新のない機体情報を破棄
        if ts - tr.last_seen_ts > self.track_ttl_s:
            del self.tracks[hexid]
```


これはメモリ上限の保証だけでなく、第1章のAUC算出において「どの入力に対してどの $n_{\\text{used}}$ が出るか」を再現可能にするための決定的な設計です。

## ネットワーク転送を PLAO に持たせない理由

ログ転送は `rsync` に任せ、PLAO自体には転送機能を持ちません。

```bash:ops/sync/sync_plao_pos_to_arena.sh
# ops/sync/sync_plao_pos_to_arena.sh より抜粋
rsync -av --append-verify \
  "${PI_USER}@${PI_HOST}:${PI_DIR}/${TODAY_FILE}" \
  "${ARENA_DIR}/"
```


転送は「失敗するもの」です。ロガーの責務は「ローカルに正しく書く」ことに閉じ、複雑な失敗処理は `rsync` という枯れた道具に任せる。これが疎結合設計の極致です。

## まとめ

**「何を測るか」を誤れば統計は無意味になりますが、「正しく貯まっていない」ログで行う統計は、それ以上に危険な誤解を招きます。**

評価指標の信頼性を、ログ基盤の信頼性で下支えする。壊れやすい環境で壊れにくい構造を選び、影響を最小化する。この「誠実な証人」としてのログ基盤が整って初めて、高度な統計解析が可能になります。
